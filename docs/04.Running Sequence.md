## System Startup, Operation, and Shutdown Narrative for `sophia_learner`

This narrative describes the complete lifecycle of the `sophia_learner` framework – from prerequisites through normal operation to graceful or error‑induced shutdown. All references to functions, classes, and modules are based on the **Exhaustive Description** and the **JSON function‑prompt table**.

---

### 1. System Pre‑requisites (Before Launch)

| Category | Requirement |
|----------|-------------|
| **Hardware** | x86_64; 4 GB RAM (8 GB recommended); 1 TB SSD (90% free for quarantines & AI training data) |
| **Operating System** | Linux (Ubuntu 20.04 / 22.04 / Debian 11+ / RHEL 8+) |
| **Python** | 3.10 or 3.11 |
| **System Packages** | `antiword`, `catdoc`, `libmagic1`, `clamav` (optional), `cron` (optional) |
| **Python Libraries** | See `requirements.txt` (watchdog, PyYAML, pydantic, python‑docx, openpyxl, xlrd, pdfplumber, PyPDF2, httpx, rich, click, etc.) |
| **AI Backend** | Ollama (installed and running locally) **or** Hugging Face `transformers` (with PyTorch) |
| **Permissions** | Read access to watched folders; write access to `data/`, `logs/`, `quarantine/`; ability to create systemd service (optional) |
| **Disk Space** | At least 5 GB free for logs + SQLite + training output + quarantine (plus space for original files) |
| **Network** | Localhost only (no external API calls except optional Ollama HTTP on 11434) |

---

### 2. User Launch Command

```bash
# From the sophia_learner project root
python -m sophia_learner.cli start --config /path/to/config.yaml

# Or as a systemd service (after installation)
sudo systemctl start sophia-learner
```

The CLI entry point is `cli/commands.py:main()` which parses the `start` command and calls `start(args)`.

---

### 3. Sequence of Loading (Startup)

#### 3‑a. Logger Initialisation
- `utils/logger.py:setup_logging(config.logging)` is called.
- Creates rotating file handlers:
  - `sophia_learner.log` (general)
  - `security.log` (security events)
  - `errors.log` (exceptions)
- Installs console handler with `rich` formatting if attached to a TTY.
- Returns a root logger and a dedicated security logger (`get_security_logger()`).

#### 3‑b. Configuration Loading
- `config/settings.py:get_config()` (singleton) calls `load_config()`.
- Reads YAML from `config/config.yaml` (or custom path).
- `config/schema.py:validate_config()` validates structure using Pydantic `ConfigSchema`.
- Calls `validate_ai_backend()` to check that the specified backend (Ollama / Transformers) is available.
- Returns a `Settings` dataclass instance with nested config objects.

#### 3‑c. Database Initialisation
- `db/database.py:Database` singleton is instantiated with `db_path = config.database.path` (default `data/sophia.db`).
- `db/database.py:connect()` creates the SQLite file and enables WAL mode.
- `db/models.py:create_tables(db)` is called – creates all tables (`files`, `versions`, `processing_logs`, `conflicts`, `metrics`, `schedule_logs`).
- If a schema migration is needed, `migrate_schema()` runs incrementally (reads `_schema_version` table).

#### 3‑d. Security Layer Setup
- `security/sandbox.py:Sandbox` initialised with limits from `config.security`.
- `security/scanner.py:VirusScanner` initialised (if `enable_virus_scan` is `True`).
- Quarantine directories are created via `processor/quarantine.py:Quarantine.__init__` (subdirs: `incoming/`, `processing/`, `processed/`, `rejected/`, `conflicts/`).

#### 3‑e. Output Management
- `ai/training_formatter.py:TrainingFormatter` created with `config.ai.output_schema`.
- `output/rotator.py:Rotator` created (size, daily rotation, compression settings).
- `output/writer.py:OutputWriter` instantiated – opens the current JSONL file for atomic appending.

#### 3‑f. Tracking Objects
- `db/file_tracker.py:FileTracker(db)` – for CRUD operations on file records.
- `db/version_tracker.py:VersionTracker(db)` – for version chains and conflict tracking.

#### 3‑g. Scheduler Core
- `scheduler/time_window.py:TimeWindow` built from `config.scheduler.processing_window` and `timezone`.
- `scheduler/rate_limiter.py:RateLimiter` with `delay_between_files_seconds`.
- `watcher/scheduler.py:ProcessingScheduler` composed of the above two.
- `scheduler/backfill.py:BackfillProcessor` created (uses `FileTracker` and `version_detector`).

#### 3‑h. AI Client
- Based on `config.ai.backend`:
  - **Ollama**: `ai/ollama_client.py:OllamaClient` connects to `http://localhost:11434` and verifies model existence via `health_check()`.
  - **Transformers**: `ai/transformers_client.py:TransformersClient` loads the model (may take several seconds and 2‑4 GB RAM). Health check verifies that the model is loaded on the specified device (CPU/CUDA).
- `ai/prompt_templates.py:PromptTemplate` loads the configured template file.

#### 3‑i. Parser Registry
- `parser/parser_registry.py:ParserRegistry.auto_register_builtins()` registers:
  - `.doc` → `DocParser`
  - `.docx` → `DocxParser`
  - `.xls` → `XlsParser`
  - `.xlsx` → `XlsxParser`
  - `.pdf` → `PdfParser`

#### 3‑j. Processing Core
- `processor/file_processor.py:FileProcessor` instantiated with all the above dependencies (trackers, AI client, writer, sandbox).

#### 3‑k. Watcher and Debouncer
- `watcher/debouncer.py:Debouncer` created with `hold_hours = config.watcher.hold_hours` (default 24).
- `watcher/event_handler.py:SophiaEventHandler` created (passes events to debouncer).
- `watcher/directory_watcher.py:DirectoryWatcher` created with the event queue.
- For each folder in `config.watcher.watch_folders`, `add_watch()` is called – starts an `inotify` observer.

#### 3‑l. Backfill (if enabled)
- If `config.watcher.backfill_on_startup` is `True`, `scheduler/backfill.py:BackfillProcessor.run_backfill()` scans all watched folders, filters files already processed or quarantined, and adds them to the DB with `status='pending'`.

#### 3‑m. Conflict Resolution Initialisation (if auto mode)
- If `config.management.conflict_resolution == "auto_keep_latest"`, `processor/conflict_resolver.py:ConflictResolver` is set to auto‑resolve version conflicts by keeping the highest version.

#### 3‑n. Start Main Loop
- The main thread enters a loop:
  - It first calls `ProcessingScheduler.wait_for_window()` – sleeps until the daily processing window begins.
  - Once inside the window, it periodically polls:
    - The event queue (from `DirectoryWatcher`) for new files.
    - The `FileTracker.get_pending_files()` for files that are due (including backfilled files).
  - For each file:
    - The `RateLimiter` enforces the configured delay.
    - `FileProcessor.process()` is called (see **Operation** below).
  - The loop exits only on shutdown signal.

### 4. Normal Operation (Per‑File Processing)

When a file is detected or picked from the pending queue, the following pipeline runs inside `FileProcessor.process()`:

| Step | Action | Component Used |
|------|--------|----------------|
| **1** | Validate file (MIME, magic bytes, filename safety) | `security/validator.py:validate_file_header()`, `check_mime_type()`, `validate_filename()` |
| **2** | Check if already processed (by SHA256) | `db/file_tracker.py:file_exists()` |
| **3** | Detect version from filename | `processor/version_detector.py:detect_version()` |
| **4** | Check for version conflicts (if multiple versions of same logical file exist) | `db/version_tracker.py:detect_conflict()` → if conflict and mode is manual, file status set to `conflicting` and user gets notified. Processing stops. |
| **5** | Move file to quarantine `processing/` | `processor/quarantine.py:move_to_quarantine(stage="processing")` |
| **6** | Get parser from `ParserRegistry` | `parser/parser_registry.py:get_parser_for_file()` |
| **7** | Extract text inside sandbox | `security/sandbox.py:run_in_sandbox(parser.extract_text, file_path)` – enforces CPU time, memory, and timeout limits. |
| **8** | Sanitise extracted text | `security/sanitizer.py:sanitize_text()`, `remove_embedded_scripts()`, etc. |
| **9** | Send to AI client (Ollama/Transformers) | `ai/ollama_client.py:process_text()` or `transformers_client.process_text()` → returns list of training samples (JSON). |
| **10** | Format and write output | `ai/training_formatter.py:validate_sample()` + `to_jsonl()` → `output/writer.py:append()` (atomic, thread‑safe). |
| **11** | Update database status to `processed` | `db/file_tracker.py:update_file_status()` |
| **12** | Optionally move original file to quarantine `processed/` | `processor/quarantine.py:mark_processed()` |
| **13** | Log metrics | `output/metrics.py:log_sample()`, `increment_counter()` |

If any step fails (e.g., parser error, AI timeout, disk full), the file status is set to `failed`, the retry counter is incremented, and the error is logged. After 3 retries, the file is moved to `quarantine/rejected/`.

### 5. Shutdown Possibilities

Shutdown can be **graceful** (user‑requested) or **error‑based** (unhandled exception, resource exhaustion, or signal).

#### 5‑a. Graceful Shutdown (User‑Initiated)

| Method | Mechanism |
|--------|-----------|
| **CLI command** | `python -m sophia_learner.cli stop` → sends SIGTERM to the main process. |
| **Systemd** | `sudo systemctl stop sophia-learner` → sends SIGTERM. |
| **Keyboard interrupt** | `Ctrl+C` in terminal → raises `KeyboardInterrupt`. |

**Graceful shutdown sequence** (handled by signal handler or `finally` block):
1. Stop accepting new events from `DirectoryWatcher` (unregister observers).
2. Wait for any currently processing file to finish (max grace period `config.management.grace_period_seconds`, default 60).
3. Cancel all pending debouncer timers (`Debouncer.cancel_file()` for all tracked files).
4. Flush and close the `OutputWriter` – final rotation and compression.
5. Close the database connection (`Database.close()`).
6. Call `Sandbox.cleanup_temp_dir()` for any leftover extraction directories.
7. Write final metrics report (optional).
8. Exit with code 0.

#### 5‑b. Error‑Based Shutdown (Unplanned)

| Error / Condition | What Happens | System State After |
|-------------------|--------------|--------------------|
| **Unhandled exception in main loop** (e.g., `ConfigError`, `DatabaseError`, `ParserError` not caught at top level) | The exception is logged, a crash dump is written, and the process exits with code 1. | May leave temporary files; database may be in an inconsistent state (but SQLite rollback will revert uncommitted transactions). Quarantine files may remain in `processing/`. |
| **Disk full** (no space to write output JSONL or database) | `OutputWriter.append()` or `Database.execute()` raises `sqlite3.OperationalError` (disk full). The main loop catches this, logs a critical error, and pauses all processing. The system does **not** shut down; it waits and retries every 60 seconds. If disk remains full for more than 1 hour, it initiates a graceful shutdown. | Paused state; no new files processed. |
| **AI backend unreachable** (Ollama down / Transformers OOM) | The AI client raises `AIConnectionError`. The `FileProcessor` retries the file (up to 3 times). After all retries fail, the file is marked `failed`. The system continues with other files. | No shutdown; degraded operation. |
| **Sandbox timeout / memory limit exceeded** | `run_in_sandbox()` raises `SandboxError`. The file is moved to quarantine `rejected/`. Process continues. | No shutdown. |
| **Kernel OOM killer** (system out of memory) | The OS kills the process without warning. | Abrupt termination – database may need `vacuum` on next start; any half‑written JSONL file may be corrupted (but atomic write prevents corruption of the last sample). |
| **Power loss / system crash** | Unsaved data in SQLite WAL is lost; JSONL file may have incomplete last line. On restart, `Database.vacuum()` and `OutputWriter.rotate()` (if file is incomplete) are triggered. | Automatic recovery possible; risk of losing the last file being processed. |

### 6. Potential Lock‑downs & Concerns Before Launch

Before deploying `sophia_learner` in production, the following **lock‑down scenarios** must be addressed to avoid system stalls, data loss, or security breaches.

| Concern | Description | Mitigation / Check |
|---------|-------------|---------------------|
| **No AI backend available** | If `config.ai.backend = "ollama"` but Ollama is not running or the specified model is missing, `health_check()` fails during startup. The system refuses to start. | Pre‑start script: `ollama list` and `curl http://localhost:11434/api/tags`. |
| **Invalid time window** | User sets `processing_window: {start: "07:00", end: "07:00"}` (same). `TimeWindow` will think the window is 0 seconds long – no processing ever occurs. | `schema.py:validate_time_window()` must reject `start == end`. |
| **Hold hours = 0** | If `hold_hours = 0`, the debouncer releases files immediately, potentially triggering reprocessing of files that are still being written. | Config validation should enforce `hold_hours >= 1` (or document that 0 disables hold). |
| **Untrusted file parsers** | Even with sandboxing, complex parsers (`antiword`, `catdoc`, `xlrd`) have had CVEs. A malicious file could escape the sandbox if resource limits are misconfigured. | Use `max_memory_mb = 256`, `max_cpu_seconds = 30`. Run parsers in separate processes (multiprocessing). Regularly update system packages. |
| **Micro‑services port conflict** | The management app (if enabled) tries to bind to `localhost:8000`. If another service uses that port, the management app fails but the watcher continues. | Use a random high port if default is busy, or disable management app. |
| **Database WAL file grows indefinitely** | Under heavy load, `-wal` and `-shm` files can become large and never checkpoint if not enough idle time. | `Database.vacuum()` scheduled daily; `checkpoint` after every 1000 writes. |
| **Quarantine directory fills up** | If no cleanup policy is set, quarantine can consume all remaining disk space. | Configure `quarantine_cleanup_days` (default 30) and run `Quarantine.cleanup_old_files()` via a cron job or internal scheduler. |
| **Rate limiter blocks forever** | If `delay_between_files_seconds` is set to a very large value (e.g., 1e9) and many files are pending, the system will appear hung. | Cap value at 3600 seconds (1 hour). |
| **Backfill of millions of files** | On first startup, `backfill` may try to insert all existing files into the database, causing memory spike and long startup time. | Add `max_files` parameter (default 1000). Use batching. Show progress. |
| **File version detection regex catastrophic backtracking** | Malformed filenames like `file_____________v1.2.3 ...` could cause ReDoS. | Use `regex` library with timeout or pre‑compile patterns with safe matching. |
| **Simultaneous file creation storm** | `watchdog` may generate thousands of `on_created` events in a second. Debouncer uses one timer per file – could exhaust thread limit. | Use a single global timer that scans a set of pending files every minute, instead of per‑file timers (alternative design). Current design mentions per‑file `threading.Timer` – rename to use `sched` or background thread with sleep. |
| **Output JSONL file corruption on crash** | Atomic write protects the last appended sample, but the entire file may be truncated if rotation occurs during a crash. | Use `OutputWriter.flush()` after every sample and after rotation. Keep integrity checks on startup. |
| **Lack of read permissions on watched folder** | `add_watch()` fails silently or raises permission error. The folder is simply not watched, but user is not notified. | Emit a critical log on `add_watch` failure and continue with remaining folders. |
| **AI output schema mismatch** | Newer versions of the training data require different fields; old samples become unreadable by training scripts. | Use `future_proof.md` strategy: include a `$schema` field and support schema versioning. |
| **Systemd service does not stop gracefully** | If `ExecStop` is not defined, `systemctl stop` kills the process immediately, leaving quarantined files in `processing/`. | Provide a proper `sophia-learner.service` file with `ExecStop=...` sending SIGTERM and waiting. |
| **Time zone shifts (DST)** | The `TimeWindow` using `pytz` may incorrectly compute next window start when clocks spring forward/fall back. | Use `zoneinfo` (Python 3.9+) and implement DST‑aware transition handling. |

### 7. Final Recommendations Before First Launch

1. **Run a dry‑run** with a small test folder containing harmless `.txt` and `.pdf` files.
2. **Enable verbose logging** (`log_level: DEBUG`) for the first day.
3. **Monitor disk space** – set up an alert for when `quarantine/` or `data/AI‑Training/` exceed 80% of available space.
4. **Set sane resource limits**:
   - `sandbox.max_file_size_mb = 20`
   - `sandbox.max_extraction_time_seconds = 15`
   - `sandbox.max_memory_mb = 256`
5. **Use `auto_keep_latest`** for version conflicts during initial testing; switch to `manual` when confident.
6. **Backup the SQLite database** daily (`Database.backup()`) – it contains the entire processing state.
7. **Test shutdown** with `kill -TERM` and verify that no file stays in `processing/` longer than the grace period.

Once these checks pass, `sophia_learner` can be launched as a background daemon and will run indefinitely, processing documents autonomously and generating AI training data.
